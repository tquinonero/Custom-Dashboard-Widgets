### NEW BUGS (fix these first)

1. **`TasksWidget.js` — `fetchUsers()` is never called (CRITICAL — users dropdown always empty)**
   File: src/components/TasksWidget.js, `useEffect` (~line 24)
   Problem: `fetchUsers` is destructured from `useDispatch('cdw/store')` but is never actually
   invoked. The store's `users` array is always the empty initial value `[]`, which means:
   - The assignee `<select>` dropdown is permanently hidden (it renders only when `users.length > 0`).
   - The "Assigned To" column always falls back to 'Unknown' for any task that has a `created_by`.
   This makes the entire user-assignment feature non-functional.
   Fix: Add `fetchUsers()` to the existing `useEffect` that already calls `fetchTasks()`:
     useEffect(() => {
         fetchTasks();
         fetchUsers();
     }, []);

2. **`store.js` — `DEFAULT_STATE.isLoading` is missing the `users` key**
   File: src/data/store.js, `DEFAULT_STATE` object (~line 27)
   Problem: The `isLoading` map is initialized with keys for `stats`, `tasks`, `posts`, `media`,
   `updates`, and `settings`, but not `users`. The `isLoading('users')` selector therefore returns
   `undefined` (not `false`) before the first `setLoading('users', ...)` dispatch, making the
   loading state unreliable on first render and potentially causing unexpected UI behavior in any
   component that checks `isLoading('users')`.
   Fix: Add the missing key to `DEFAULT_STATE.isLoading`:
     isLoading: {
         stats: false,
         tasks: false,
         posts: false,
         media: false,
         users: false,      // <-- add this
         updates: false,
         settings: false,
     },

3. **`QuickLinksWidget.js` — `useDispatch` imported but unused; `users` selected but never rendered**
   File: src/components/QuickLinksWidget.js (~lines 1-4)
   Problem: The component imports `useDispatch` from `@wordpress/data` but never calls it.
   It also selects `users` from the store (`useSelect(... getUsers())`) but never references
   `users` anywhere in the returned JSX. The dead `useSelect` subscription means QuickLinksWidget
   re-renders every time the `users` array changes in the store (e.g. when TasksWidget loads
   users), for no reason. This will also trigger an ESLint `no-unused-vars` error.
   Fix: Remove both the `useDispatch` import and the `users` selector. Since `useSelect` is only
   used for `getUsers()`, remove the entire import and selector:
     // Remove this line entirely:
     import { useSelect, useDispatch } from '@wordpress/data';
     // Remove this line entirely:
     const users = useSelect((select) => select('cdw/store').getUsers());

4. **`CommandWidget.js` — bare `cdwData.nonce` references lack `window.` and optional chaining**
   File: src/components/CommandWidget.js (~lines 31, 48, 116, 196)
   Problem: All four `apiFetch` calls in this component pass:
     headers: { 'X-WP-Nonce': cdwData.nonce }
   `cdwData` is accessed as a bare global without `window.` or optional chaining (`?.`).
   If `cdwData` is not defined (e.g. in a test environment, or if `wp_localize_script` fails),
   this throws `ReferenceError: cdwData is not defined`, crashing the entire component on mount.
   Every other component in the codebase uses `window.cdwData?.adminUrl || ''` consistently.
   Fix: Replace all four occurrences with:
     headers: { 'X-WP-Nonce': window.cdwData?.nonce }

5. **`save_settings()` PHP — settings can be set but never cleared from the UI**
   File: includes/class-cdw-rest-api.php, `save_settings()` (~line 565)
   Problem: Every field in `save_settings()` is guarded so that empty/zero values are silently
   ignored and the stored option is never updated:
   - `email`:    `if ( isset(...) && ! empty(...) )` — empty string is skipped; old value stays.
   - `docs_url`: same guard — cannot clear the docs URL.
   - `font_size`: `absint('')` returns 0, which fails `$size >= 10` — cannot clear the font size.
   - `bg_color`, `header_bg_color`, `header_text_color`: `validate_hex_color('')` returns `''`,
     which fails the `$color !== ''` guard — color options cannot be cleared.
   Once a user saves any of these values, there is no way to wipe them from the settings form.
   Fix: For each field, explicitly handle the empty case by deleting the option when the submitted
   value is blank, rather than silently skipping the update. Example for email:
     if ( isset( $settings['email'] ) ) {
         if ( ! empty( $settings['email'] ) ) {
             $email = $this->validate_email( $settings['email'] );
             if ( $email ) {
                 $email = sanitize_email( $email );
                 update_option( 'cdw_support_email', $email );
                 update_option( 'custom_dashboard_widget_email', $email );
             }
         } else {
             delete_option( 'cdw_support_email' );
             delete_option( 'custom_dashboard_widget_email' );
         }
     }
   Apply the same pattern for `docs_url` (delete when empty), `font_size` (delete when empty/0),
   and the three color options (delete when empty string is submitted).
