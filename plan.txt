## Suggested new commands

### HIGH VALUE — missing and very commonly needed

~~1. rewrite flush~~ ✓ implemented

2. email test <address>
   - Diagnose SMTP/mailer configuration without leaving the dashboard.
   - Calls wp_mail() with a simple test body and reports success/failure.
   - Broken email is a top-3 support issue on every WP site.
   - readonly: false, destructive: false

~~3. core version~~ ✓ implemented

~~4. comment list/approve/spam/delete~~ ✓ implemented

5. db optimize
   - Run OPTIMIZE TABLE across all tables. Common monthly maintenance to reclaim
     fragmented space after mass deletes.
   - Implementation: $wpdb->get_results('SHOW TABLES') loop + $wpdb->query('OPTIMIZE TABLE …')
   - readonly: false, destructive: false

---

### MEDIUM VALUE — useful for power users

6. user reset-password <username|id>
   - Generate a password reset link without going to the Users screen.
   - APIs: get_password_reset_key(), network_site_url('wp-login.php?action=rp')
   - Output the full reset URL (show-once).
   - readonly: false, destructive: false

~~7. media list [<count>]~~ ✓ implemented

8. post count [<type>]
   - Quick wp_count_posts($type) showing breakdown by status
     (publish / draft / trash / pending).
   - Instant health check for large sites.
   - readonly: true

~~9. plugin update-all~~ ✓ implemented

10. db repair
    - Run REPAIR TABLE on all tables — useful after a crash.
    - Same pattern as db optimize.
    - readonly: false, destructive: false

11. db check
    - Run CHECK TABLE on all tables and report errors.
    - Pre-flight check before a migration or major upgrade.
    - readonly: true

---

### LOWER PRIORITY — nice to have

12. term list <taxonomy>
    - get_terms(['taxonomy' => …]) — debug taxonomy issues (missing categories, orphaned tags)
    - readonly: true

13. user count
    - count_users() — totals by role.
    - One-liner diagnostic for auditing who has admin access.
    - readonly: true

14. post trash <id>
    - wp_trash_post() — safer alternative to post delete (sends to trash, not hard delete)
    - readonly: false, destructive: false

15. site url
    - Display siteurl and home options side by side.
    - First thing to check after a migration/staging clone to find URL mismatches.
    - APIs: get_site_url(), get_home_url()
    - readonly: true

---

## NOT recommended

- db query <sql>  — Even read-only SQL execution via a dashboard widget is a significant
  injection risk surface. The existing db size / db tables / search-replace cover the safe subset.

- media regenerate — Long-running (timeouts on large libraries), requires chunking. Better
  handled by a dedicated plugin (Regenerate Thumbnails).

## Implementation checklist (per new command)

For each new command, three files must be updated:

1. includes/services/class-cdw-cli-service.php
   - Add case to the relevant handle_*_command() switch, or create a new handle_*_command()
     and wire it in the top-level switch in execute().
   - Update inline help text (default: case) and global help (handle_help_command).

2. includes/controllers/class-cdw-cli-controller.php
   - Add entry to get_command_definitions() in the appropriate category array.

3. includes/class-cdw-abilities.php
   - Register a new cdw/* ability with correct readonly/destructive annotations.
   - Add a build_cli_command() case to map input schema fields to the CLI string.
   - Update ability count in AbilitiesTest.php.

For AI tool coverage also update:
   - includes/services/class-cdw-ai-service.php → get_tool_definitions() array.

Tests to add (per command):
   - tests/php/unit/CliServiceHandlersTest.php — success case, error/no-args case,
     WP_Error propagation where applicable.

### Structured Page Builder (cdw/build-page)

Accepts JSON input to generate Gutenberg block markup. This is abilities-only (no CLI equivalent - JSON input cannot survive whitespace tokeniser).

**Abilities-only command (not in CLI widget or AI tools):**

Input:
```json
{
  "title": "Page Title",
  "sections": [
    {"type": "cover", "title": "Hero", "image": "url"},
    {"type": "two-column", "left": {"image": "url"}, "right": {"heading": "...", "text": "..."}},
    {"type": "footer", "columns": [...]}
  ]
}
```

Implementation:
1. New ability `cdw/build-page` in `includes/class-cdw-abilities.php`
2. New renderer file: `includes/renderers/class-cdw-section-renderers.php`
   - render_cover_section()
   - render_two_column_section()
   - render_three_column_section()
   - render_footer_section()
3. Each renderer returns Gutenberg block markup string
4. Main handler concatenates and creates/updates post
5. Returns post_id, title, section_count
